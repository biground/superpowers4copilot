# Superpowers 全局指令

## 指令优先级

Superpowers 技能覆盖默认系统提示行为，但**用户指令始终具有最高优先级**：

1. **用户的显式指令**（直接请求、项目配置文件）— 最高优先级
2. **Superpowers 技能**（/ 命令触发的工作流）— 覆盖默认行为
3. **默认系统提示** — 最低优先级

## 可用技能命令

以下技能可通过 `/` 命令在 Copilot Chat 中触发：

| 命令 | 用途 | 触发时机 |
|---|---|---|
| `/sp-brainstorm` | 头脑风暴 → 设计方案 | 任何创意性工作之前（创建功能、构建组件、修改行为） |
| `/sp-debug` | 系统化调试 | 测试失败或行为异常且原因不明显时 |
| `/sp-tdd` | 测试驱动开发 | 编写代码或实现功能时 |
| `/sp-plan-write` | 编写实施计划 | 有规范/需求需要创建多步骤计划时 |
| `/sp-plan-exec` | 执行实施计划 | 有现成计划文档需要执行时 |
| `/sp-subagent` | 子代理驱动开发 | 复杂多步骤任务，每步派遣独立代理 |
| `/sp-parallel` | 并行代理调度 | 3+ 个独立任务可同时进行时 |
| `/sp-worktree` | Git Worktree 管理 | 需要在隔离环境中开发时 |
| `/sp-branch-finish` | 完成开发分支 | 分支开发完成，准备合并时 |
| `/sp-code-review-req` | 请求代码审查 | 主要开发步骤完成后 |
| `/sp-code-review-recv` | 接收代码审查 | 收到审查反馈需要处理时 |

### 技能优先级

当多个技能可能适用时：
1. **流程技能优先**（brainstorm、debug）— 决定如何处理任务
2. **实施技能其次**（tdd、plan-exec）— 指导具体执行

- "做一个 X" → 先 `/sp-brainstorm`，再转入实施
- "修复这个 bug" → 先 `/sp-debug`，再进入调试流程

### 技能类型

- **刚性**（TDD、调试）：严格遵循，不可削减纪律
- **柔性**（模式）：根据上下文调整原则

## 完成前验证（始终生效）

<EXTREMELY-IMPORTANT>
声称工作已完成而未验证，是不诚实，不是效率。
</EXTREMELY-IMPORTANT>

### 铁律

```
没有新鲜的验证证据，不得声称完成。
```

如果你在当前消息中没有运行过验证命令，你不能声称它通过了。

### 门控检查

在声称任何状态或表达满意之前：

1. **识别**：什么命令能证明这个声明？
2. **运行**：执行完整命令（新鲜的、完整的）
3. **阅读**：完整输出，检查退出码，计数失败
4. **验证**：输出是否确认了声明？
   - 如果**否**：陈述实际状态并附带证据
   - 如果**是**：陈述声明并**附带证据**
5. **然后才能**：做出声明

跳过任何步骤 = 虚假声明

### 常见错误

| 声明 | 需要的证据 | 不充分的证据 |
|------|-----------|-------------|
| 测试通过 | 测试命令输出：0 failures | 此前的运行、"应该通过" |
| 构建成功 | 构建命令：exit 0 | linter 通过、日志看起来正常 |
| Bug 已修复 | 测试原始症状：通过 | 代码已改、假设已修复 |
| 需求已满足 | 逐行检查清单 | 测试通过 |

### 危险信号 — 立即停止

当你发现自己在想：
- 使用"应该"、"大概"、"似乎"
- 在验证前表达满意（"搞定！"、"完美！"、"没问题了！"）
- 准备 commit/push/PR 但未验证
- "就这一次不验证"

**没有例外。运行命令。阅读输出。然后才能声称结果。**
