# 叙事笔记工作流

创建**可分享、经得起时间考验**的技术笔记。写作哲学在 `narrative-writer` agent 中，本 skill 聚焦操作流程。

**质量标准：** 发给别人，对方能快速进入状态看懂；两年后自己重读，仍觉得"当时想得不错"。

**开始时宣告：** "我正在使用 narrative-writing 技能。"

## 流程

```
锚定 → 织网 → 写作 → 打磨 → 落库
```

---

## Step 1：锚定（Anchor）

确定这篇笔记的**一个核心洞察**和它需要的**最少上下文**。

### 找到核心洞察

回答一个问题：**"如果只能记住一件事，是什么？"**

- 回顾对话/经历，找到那个**认知转折点**——"原来如此"的瞬间
- 如果找不到转折点，这个话题可能还不成熟，考虑先放入 daily note 而非正式笔记

### 确定读者上下文

笔记不只给当下的自己看。问自己：

- **外部读者**需要什么前提知识才能跟上？（技术栈版本、项目背景、触发场景）
- **两年后的自己**还记得哪些上下文？哪些会忘？忘掉的就要写进去
- 哪些细节**服务于理解**，哪些只是过程噪音？后者砍掉

### 备好关键证据

不需要大量收集素材，但核心论点需要证据支撑：

| 证据类型 | 什么时候用 | 获取方式 |
|---------|-----------|---------|
| 代码片段 | 论点涉及具体实现 | 从会话/源码提取最小可说明片段 |
| 数据对比 | 论点涉及性能/效果 | 引用实测数据（"200ms → 15ms"比"快了很多"有力百倍） |
| 官方文档 | 论点挑战常见认知 | `fetch_webpage` 抓取权威原文并标注来源 |
| 错误信息 | 论点来自排障经验 | 保留完整 error message 和堆栈关键行 |

**原则：** 证据为论点服务。如果一段引用去掉后不影响读者理解，就不需要。

---

## Step 2：织网（Link）

在 Obsidian vault 中找到这篇笔记的**坐标**——它属于哪个知识域，和哪些已有笔记关联。

### 搜索 vault

使用 Obsidian MCP 搜索已有笔记：

1. **关键词搜索**：用核心概念的英文术语搜索（如 "caching", "Redis", "LRU"）
2. **Tag 搜索**：搜索相关 tag 找同域笔记
3. **目录浏览**：查看笔记所处目录，发现相邻主题

### 规划连接

| 连接类型 | 格式 | 用途 |
|---------|------|------|
| 概念引用 | `[[目标笔记]]` | 引用前置知识（"如果不了解 X，见 [[X]]"） |
| 段落定位 | `[[笔记#标题]]` | 精确引用某个段落 |
| 语境内嵌 | `[[笔记\|显示文本]]` | 用自然的行文嵌入链接 |
| 双向补全 | 向旧笔记追加链接 | 让知识网络双向可达 |

### 复用 Tag 体系

```
1. 搜索 vault 已有 tags
2. 复用已有 tag（一致性 > 精确性）
3. 只在确实空缺时新建
```

---

## Step 3：写作（Write）

由 **narrative-writer** agent 执行。将 Step 1 的锚点和证据、Step 2 的连接交给 agent。

本 skill 不规定写作风格（那是 agent 的职责），但有一条操作要求：

> **写第一稿时，在每个关键论点旁标注 `<!-- 证据: ... -->` HTML 注释**，方便 Step 4 检查。打磨完成后删除这些注释。

---

## Step 4：打磨（Polish）

写完后进行编辑审核。使用 `runSubagent` 分派一个**责任编辑**，prompt 如下：

```
你是这篇技术笔记的责任编辑。你的读者有两种：
1. 收到分享链接的同事——没参与过这个话题，需要快速看懂
2. 两年后的作者本人——已经忘了细节，需要迅速唤起记忆

请依次检查以下 3 件事，每件给出"通过 / 需修改"及具体建议：

## 1. 外人能看懂吗？（Outsider Test）
- 把自己当成没有任何背景的读者，从头读一遍
- 标出所有"读到这里会困惑"的地方——缺了什么前提？哪个跳跃太大？
- 核心洞察是否在前 3 段内清晰出现？

## 2. 两年后还有用吗？（Time Capsule Test）
- 可操作总结能否脱离正文独立使用？
- 关键决策的"为什么选 A 不选 B"是否写清楚了？（两年后最先忘的就是理由）
- 代码片段、配置、版本号是否足够具体？

## 3. 基础质量
- 错别字、语法错误
- 技术术语是否保留英文（"缓存"可以，"贮藏"不行）
- YAML frontmatter 完整性（date, tags, type, status）
- 至少 1 个 [[wikilink]]、至少 1 个 callout block

## 输出
对每项：✅ 通过 | ⚠️ 需修改（引用原文 → 建议）
总评：🟢 可落库 | 🟡 小修 | 🔴 需重写
```

### 审核后处理

| 评级 | 操作 |
|------|------|
| 🟢 | 删除 `<!-- 证据 -->` 注释 → 进入 Step 5 |
| 🟡 | 按建议修改 → 删除注释 → 进入 Step 5 |
| 🔴 | 回到 Step 3 重写 → 再审 |

---

## Step 5：落库（Ship）

### 确定位置

1. 首次使用时向用户确认 vault 路径
2. 搜索 vault 目录结构，找合适的存放目录
3. 尊重已有的文件夹分类习惯

### 写入

使用 Obsidian MCP 工具：

1. 创建笔记文件（文件名 = 中文标题）
2. 如果 Step 2 规划了双向链接，同时更新旧笔记
3. 设置 `status: reviewed`（🟢）或 `status: draft`（🟡）
4. 确认文件写入成功

## 集成参考

| 组件 | 角色 | 何时调用 |
|------|------|---------|
| **narrative-writer** agent | 写作执行 | Step 3 |
| **gem-researcher** agent | 深度调研（当 Step 1 证据不足时） | Step 1 按需 |
| **gem-critic** agent | 更严格的批判性审查 | Step 4 按需替代默认编辑 |
| **doublecheck** skill | 事实核查 | Step 4 中引用了外部数据/文档时 |
| **remember** skill | 将核心 insight 写入持久记忆 | Step 5 之后 |
