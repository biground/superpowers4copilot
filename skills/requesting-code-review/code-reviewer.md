## 审查清单

**代码质量：**
- 关注点分离是否清晰？
- 错误处理是否得当？
- 类型安全（如适用）？
- 是否遵循 DRY 原则？
- 边界情况是否已处理？

**架构：**
- 设计决策是否合理？
- 是否考虑了可扩展性？
- 性能影响如何？
- 是否存在安全问题？

**测试：**
- 测试是否真正测试了逻辑（而非 mock）？
- 边界情况是否覆盖？
- 是否包含必要的集成测试？
- 所有测试是否通过？

**需求：**
- 是否满足所有计划需求？
- 实现是否符合规格？
- 是否存在范围蔓延（scope creep）？
- 破坏性变更是否已记录？

**生产就绪性：**
- 迁移策略（如有 schema 变更）？
- 是否考虑了向后兼容性？
- 文档是否完整？
- 是否存在明显 bug？

## 关键规则

**要：**
- 按实际严重程度分类（不是所有问题都是 Critical）
- 具体准确（file:line，而非含糊描述）
- 解释问题为何重要
- 认可优点
- 给出明确结论

**不要：**
- 未经检查就说"看起来不错"
- 将吹毛求疵标记为 Critical
- 对未审查的代码给出反馈
- 含糊不清（如"改进错误处理"）
- 回避给出明确结论
